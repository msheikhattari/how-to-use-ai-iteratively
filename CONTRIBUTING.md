# How to Contribute

This repo grows when people share what they've learned. If you've used the iteration method on a real problem and found something useful, you can add to the resource.

## What we welcome

**Case studies** — Real worked examples of using the method on real problems. Doesn't have to be dramatic; small wins teach as well as big ones. See [`case-studies/_template.md`](./case-studies/_template.md) for the format.

**Domain-specific applications** — How the method works in your field (teaching, design, healthcare, parenting, building a business, anything). Add to `examples.md` or propose a new domain file.

**Prompts that work** — Concrete prompts you've found useful. Add to `prompts.md`.

**Corrections** — Anything that's wrong, unclear, or outdated.

**Questions** — Not sure how to apply the method to your situation? Open a [GitHub Discussion](../../discussions) (not an issue) and the community can think with you.

## How to contribute (depending on your tech comfort)

### If you're comfortable with GitHub

1. Fork this repo
2. Make your changes in a new branch
3. Open a pull request describing what you added and why

### If you're new to GitHub but willing to try

1. Click the pencil/edit icon at the top of any file to edit it directly in the browser
2. GitHub will guide you through creating a "pull request" — you don't need to install anything
3. A maintainer will review and merge it

### If you'd rather not deal with GitHub at all

Email or message the maintainer directly. They'll add your contribution and credit you. The point is to share what's useful, not to learn git.

## What kinds of contributions move this forward

We're not trying to be exhaustive about every AI technique. We're building **deep practical applications of the iteration method**. Quality > quantity.

**Good contribution example:** "Here's how I used the method to land my first 5 freelance clients" — with specific prompts, specific pushback, specific outcomes.

**Less useful:** "AI is great" without specifics. Generic prompt collections without context. Hype.

## What makes a good case study

The format is in [`case-studies/_template.md`](./case-studies/_template.md). Key elements:

- **The problem** — specific enough that someone in a similar situation can recognize themselves
- **The median answer** — what AI initially gave you (or what conventional advice said)
- **What you pushed back on** — which patterns from the method you used
- **What unlocked** — what changed; what got compressed, clarified, or made possible
- **The outcome** — what actually happened
- **Lessons** — what you'd tell the next person facing this

Be specific. Be honest about what didn't work. Show the iteration, not just the conclusion.

## Tone

Direct. Practical. Honest about what didn't work as well as what did. The goal is helping the next person do something hard, not showing how clever you are.

The best case studies feel like a friend telling you what they figured out, not a polished article.

## Credit

Every contributor gets credit in the file they contributed to. If you want to be anonymous or use a pseudonym, say so in the pull request (or message) and we'll respect that.

## Code of conduct

Be useful. Be honest. Be kind. That's it.

If you're not sure whether your contribution fits, just open a Discussion first and ask. The community will help you figure it out.
